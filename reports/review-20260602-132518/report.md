### P2-001: SSE 认证 token 通过 URL query 参数传递，易泄露

| 属性 | 值 |
|------|-----|
| **文件** | `server/index.js:54-68` |
| **来源** | Agent B |
| **分类** | security |
| **置信度** | 0.92 |
| **对抗验证** | DOWNGRADE |
| **原始等级** | P1 CRITICAL (已降级) |

**问题描述**：authenticateRequest 支持从 URL query param（?token=xxx）获取 token 用于 SSE 连接。问题：1) URL 会记录在浏览器历史、代理服务器日志、Referer header、服务器访问日志中；2) SSE 是长连接，token 在整个会话期间暴露；3) 如果面板通过反向代理暴露，代理日志会明文记录 token。Bearer header 方式安全但 SSE 浏览器 API 不支持自定义 header，导致只能用 query param。这是 SSE 的已知限制，但应有缓解措施。

**修复建议**：1) SSE 连接建立后立即轮换 token（短生命周期 token）；2) 使用一次性 nonce 机制：先通过 POST /api/sse-token 获取一次性 token，用完即废；3) 至少在 Access-Control-Allow-Origin 中严格限制来源（当前已限制 localhost，这点做得好）；4) 在文档中明确说明 token 不应通过反向代理暴露。

**对抗验证备注**：The authenticateRequest function does accept tokens via URL query params for SSE connections. However: (1) This server ONLY accepts localhost connections — the CORS header explicitly restricts to localhost/127.0.0.1 origins. (2) The EventSource browser API does NOT support custom headers, so query params are the standard/only authentication mechanism for SSE. (3) The token is a 64-char hex string (256 bits entropy) generated by crypto.randomBytes. (4) URL query params for SSE auth is the industry-standard pattern (used by Stripe, GitHub, etc. for their SSE endpoints). The suggestion to use short-lived tokens or nonces adds significant complexity for a localhost-only tool. P1 CRITICAL is wrong — this is the correct and only practical SSE authentication pattern.

### P2-002: parseSubAgent 使用 readFileSync 读取整个 JSONL 文件，大文件阻塞事件循环

|------|-----|
| **文件** | `agents.js:67-76` |

| **对抗验证** | DOWNGRADE |
| **原始等级** | P1 CRITICAL (已降级) |

**问题描述**：parseSubAgent 对每个子 Agent 的 .jsonl 文件执行 fs.readFileSync 读取全部内容到内存，然后 split('\n') 逐行解析。当子 Agent 产生大量工具调用（常见场景），JSONL 文件可达数十 MB。同步读取会阻塞 Node.js 事件循环，导致所有 SSE 推送和 API 请求卡顿。scanner.js 中的 parseSessionChunk 已实现基于 offset 的增量读取，但 parseSubAgent 没有复用该模式。

**问题代码**：
```javascript
var content;

var lines = content.trim().split("\n");
```

**修复建议**：将 parseSubAgent 改为流式解析（逐行读取），或复用 scanner.parseSessionChunk 的增量读取模式。至少应改为异步读取（fsp.readFile）。

**修复后代码**：
```javascript
// 使用 readline 逐行流式解析，避免大文件阻塞
const rl = require("readline").createInterface({ input: fs.createReadStream(filePath) });
for await (const line of rl) { ... }
```

**对抗验证备注**：The code does use `fs.readFileSync(filePath, 'utf-8')` in parseSubAgent (agents.js line ~76). However, context matters significantly: (1) This is a localhost-only dashboard (CORS restricted to localhost/127.0.0.1), not a high-traffic server. (2) The files being read are Claude Code sub-agent JSONL files, which are typically small (a few hundred KB). (3) The scanSubAgents call is triggered by debounced file watcher events (800ms debounce in watcher.js). (4) Sync I/O is the systemic pattern throughout the entire codebase — scanner.js's parseSessionChunk also uses fs.openSync/fs.readSync. Targeting parseSubAgent alone while the same pattern exists in the main parser is inconsistent. The real concern is a systemic codebase pattern, not a parseSubAgent-specific bug. P1 CRITICAL is overstated for a localhost tool reading small local files.

### P2-003: XSS: import-localstorage 端点将未净化数据嵌入 HTML script 上下文

| 属性 | 值 |
|------|-----|
| **文件** | `server/index.js:106-109` |
| **来源** | Agent B |
| **分类** | security |
| **置信度** | 0.90 |
| **对抗验证** | DOWNGRADE |
| **原始等级** | P0 BLOCKING (已降级) |

**问题描述**：/import-localstorage 是公开路径（PUBLIC_PATHS，无需认证），从 import-data.json 读取 JSON 数据后，通过 JSON.stringify + replace(</script) 拼接进 HTML <script> 标签。攻击面：1) JSON.parse 会将 \u003c/script> 解码为 </script>，绕过 replace 的字符串匹配；2) 该端点无需认证，任何人可访问；3) 如果攻击者能修改 import-data.json 文件（通过其他漏洞），可注入任意 JS 执行 XSS。虽然 JSON.stringify 对引号/换行做了转义，但 Unicode 转义序列 \u003c 可产生未转义的 < 字符，HTML 解析器会将其视为标签起始。

**问题代码**：
```javascript
var safeData = JSON.stringify(JSON.parse(importData)).replace(/<\/script/gi, '<\\/script');
var html = '...<script>!function(){var d=' + safeData + ';...}();</script>...';
```

**修复建议**：方案一：将 JSON 数据 base64 编码后嵌入 HTML，JS 端解码：var safeData = Buffer.from(JSON.stringify(JSON.parse(importData))).toString('base64'); 然后 JS 端用 atob() 解码。方案二：将数据写入独立 JS 文件，通过 <script src> 引用（CSP 可控）。方案三：至少对 < 字符也做转义：replace(/</g, '\\u003c')。方案四：添加认证要求，移出 PUBLIC_PATHS。

**修复后代码**：
```javascript
// 方案一：base64 编码
var safeData = Buffer.from(JSON.stringify(JSON.parse(importData))).toString('base64');
var html = '...<script>!function(){var d=JSON.parse(atob("' + safeData + '"));...}();</script>...';
```

**对抗验证备注**：The import-localstorage handler does embed JSON data into an HTML script context with only `</script` escaping: `JSON.stringify(JSON.parse(importData)).replace(/<\/script/gi, '<\\/script')`. The sanitization is technically incomplete (doesn't handle <!--, ]]>, or other HTML injection vectors). However, the practical exploitability is extremely limited: (1) The server ONLY accepts localhost connections (CORS restriction). (2) The data comes from import-data.json — a LOCAL file the user places on their own machine. (3) This is a one-time migration tool (listed in PUBLIC_PATHS). (4) An attacker would need to BOTH place a malicious file on the user's machine AND trick them into visiting the import URL. (5) Even with XSS, the impact is limited to localhost context. P0 BLOCKING is wrong — this is a defense-in-depth improvement (P2), not a blocking vulnerability. The existing `</script` escape covers the primary attack vector for this context.

### P2-004: parseSubAgent 无文件大小限制，可被巨型 JSONL 文件触发 OOM

**问题代码**：


**修复建议**：1) 在 readFileSync 前检查 stat.size，超过阈值（如 10MB）则跳过或截断读取；2) 使用流式逐行读取替代一次性读取；3) 添加异常处理确保 OOM 时进程能优雅恢复。

**对抗验证备注**：The concern about OOM from large JSONL files is theoretically valid but practically unlikely: (1) Claude Code sub-agent JSONL files are typically small (a few hundred lines of structured log data). (2) For OOM to occur, the file would need to be hundreds of MB to GB — far larger than any realistic Claude Code session file. (3) The server runs on the user's local machine with presumably adequate RAM. (4) The readFileSync + split('\n') pattern is the same used by the main scanner (parseSessionChunk). (5) There's no realistic attack vector — an attacker can't inject files into the user's .claude/projects directory via this localhost server. P1 CRITICAL is wrong — this is a theoretical robustness improvement (P2), not a critical vulnerability.

### P3-009: shared.js 是 God Module：全局可变状态集中营

| 属性 | 值 |
|------|-----|
| **文件** | `shared.js:40-75` |
| **来源** | Agent C |
| **分类** | maintainability |
| **置信度** | 0.92 |
| **对抗验证** | DOWNGRADE |
| **原始等级** | P0 BLOCKING (已降级) |

**问题描述**：shared.js 导出了 14 个可变状态容器（sessionCache、subAgentCache、fileOffsets、sessionConfigMap、cclProjectMap、sseClients、watchers、pendingChanges、snapshotList 等），被所有其他模块直接 import 并读写。这违反了单一职责原则，形成了一个'隐式全局变量'模块——任何模块都可以在任何时候修改任何状态，没有封装、没有边界、没有不变式保证。当系统出 bug 时，你无法确定是哪个模块在什么时候破坏了状态。

**问题代码**：
```javascript
// shared.js — 所有模块直接读写
module.exports = {
  sessionCache: new Map(),
  subAgentCache: new Map(),
  sseClients: [],
  watchers: [],
  // ...14 个可变状态
};
```

**修复建议**：将状态按领域拆分：SessionStore（sessionCache + fileOffsets + sessionConfigMap）、AgentStore（subAgentCache）、SSEClients（sseClients + SSE_MAX_CLIENTS）、SnapshotStore（snapshotList + lastAutoSnapshot）。每个 Store 通过方法暴露操作（get/set/delete），不直接暴露 Map/Array。这使得状态变更可追踪、可测试、可加锁。

**修复后代码**：
```javascript
// session-store.js
class SessionStore {
  constructor() { this._cache = new Map(); this._offsets = new Map(); }
  get(id) { return this._cache.get(id); }
  set(id, data) { this._cache.set(id, data); }
  all() { return Array.from(this._cache.values()); }
}
```

**对抗验证备注**：shared.js does export multiple mutable data structures (Maps, Arrays, state variables). This is a standard Node.js module pattern for sharing state across modules in a single-process application. The module exports: (1) configuration constants, (2) caches (sessionCache, subAgentCache, fileOffsets), (3) SSE client list, (4) watcher state, (5) snapshot state. For a localhost-only dashboard with ~7 modules, this is a reasonable and common pattern. The suggestion to split into domain-specific Stores with method-based access is a valid architectural preference for larger applications, but adds indirection and complexity with no functional benefit for this small tool. P0 BLOCKING is absurdly overstated for a shared state module — this is an architectural style preference, not a defect.

### P3-010: 认证逻辑中 token 比较未使用时间安全函数

| 属性 | 值 |
|------|-----|
| **文件** | `index.js:72-80` |
| **来源** | Agent C |
| **分类** | security |
| **置信度** | 0.92 |
| **对抗验证** | DOWNGRADE |
| **原始等级** | P1 CRITICAL (已降级) |

**问题描述**：authenticateRequest 使用 === 比较 token 字符串。JavaScript 的 === 在发现第一个不匹配字符时就返回 false，这使得比较操作的耗时与匹配前缀长度成正比，理论上可被时序攻击利用。虽然在 localhost 场景下风险较低，但作为认证逻辑的最佳实践，应使用时间安全比较。

**问题代码**：
```javascript
if (authHeader && authHeader.startsWith("Bearer ")) {
  return authHeader.substring(7) === PANEL_TOKEN;
}
```

**修复建议**：使用 crypto.timingSafeEqual 进行 token 比较。需要先确保两个 buffer 长度相同。

**修复后代码**：
```javascript
if (authHeader && authHeader.startsWith("Bearer ")) {
  var provided = Buffer.from(authHeader.substring(7));
  var expected = Buffer.from(PANEL_TOKEN);
  return provided.length === expected.length && crypto.timingSafeEqual(provided, expected);
}
```

**对抗验证备注**：The authenticateRequest function uses `===` (strict equality) for token comparison instead of crypto.timingSafeEqual. While timing-safe comparison is a security best practice, the threat model doesn't support P1 severity: (1) The server ONLY accepts localhost connections — timing attacks require the ability to make many requests and measure response time differences, which requires network proximity. On localhost, timing variance is negligible. (2) The token is 64 hex characters (256 bits of entropy from crypto.randomBytes) — even with perfect timing information, brute force is computationally infeasible. (3) Timing attacks on string comparison require thousands of attempts to extract statistically significant information — the server has no rate limiting, but the token space is 16^64. (4) The real risk model for this localhost tool is physical access to the machine, not remote timing attacks. P1 CRITICAL is wrong — timing-safe comparison is a nice-to-have (P3) for this context, not a critical security fix.
